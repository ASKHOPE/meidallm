# Product & Usage Metrics  
- **Feature Adoption & Journeys:** Track which buttons, features or pages users click and the exact paths they take. Use event logging or analytics tools to measure completion of key workflows (e.g. onboarding). Funnel and flow analysis reveal drop-offs: e.g. track what fraction of signups finish onboarding, or which step in a multi-page form users abandon.  
- **Session Frequency & Duration:** Log when users start/end sessions. Compute Daily/Monthly Active Users (DAU/MAU) and session lengths. Spikes or lulls in usage can signal feature interest or engagement problems. For example, decreasing login frequency often precedes churn.  
- **Key Behavioral Events:** Record critical actions (e.g. “created invoice”, “sent report”) to measure Time‑To‑Value (TTV) – the time from signup to first meaningful action. Shorter TTV correlates with retention. Also measure NPS/CSAT/CES via surveys to gauge satisfaction. These quantitative metrics (activation rate, TTV, feature adoption %, retention/churn rates, NPS/CSAT) are standard product‑success indicators.  

## User Engagement & Retention Metrics  
- **Activation & TTV:** % of new users who complete onboarding or a core action (“activation”), and how long it takes. For example: if 1,000 users sign up but only 100 complete setup in one day, that flags onboarding friction.  
- **Active User Tiers:** Classify users by activity: daily, weekly, monthly. Track DAU/MAU ratios to spot engagement trends (a rising MAU/DAU ratio means more people use the app consistently). Declines can prefigure churn.  
- **Churn & Cohorts:** Compute churn rate (users lost in a period) and retention by cohort. Retention curves and survival analysis help identify when/why users leave. Cohort analytics show if new features improve stickiness.  
- **Account Expansion Triggers:** Monitor usage relative to plan limits (e.g. storage 90% full, user seats 90% used). These signals can trigger upsell offers or warnings. Growth in feature use (e.g. adding new team members) is a positive expansion signal. 

## Business & Financial Metrics  
- **Customer Acquisition Cost (CAC):** Track total sales & marketing spend divided by new customers acquired in a period. Attribute CAC down to channel or campaign level to find cost-effective acquisition paths. For example, if $10,000 was spent and 50 new accounts signed up, CAC = $200 per customer.  
- **Monthly Recurring Revenue (MRR):** Sum of all customers’ monthly subscription payments. In practice, MRR = (number of active accounts) × (average revenue per account). Exclude one-time fees (MRR is *only* recurring subscription revenue). Track new MRR, expansion MRR (upgrades), contraction MRR (downgrades), and churned MRR.  
- **Customer Lifetime Value (LTV/CLV):** The total revenue a customer will generate over their subscription lifetime. CLV is often estimated as average revenue per user (ARPU) × average customer lifespan. For instance, if a customer pays $50/mo and stays 24 months on average, CLV ≈ $1,200. LTV helps judge how much can be spent on CAC. (A healthy SaaS often targets LTV ≥ 3× CAC.)  
- **Other KPIs:** Track Annual Recurring Revenue (ARR = 12× MRR), ARPU per plan, gross & net revenue retention, LTV/CAC ratio, etc. Connecting user actions to revenue (e.g. usage-based billing) can enable forecasts of expansion and churn impact.  

## Technical & Performance Metrics  
- **Page Load & API Latency:** Measure end-to-end response times (UI render, API call round-trip). Log timing for each page load or API request. Slow responses (e.g. >2s) should be flagged. These metrics reveal performance regressions or infrastructure bottlenecks.  
- **Error, Crash & Exception Logging:** Capture all frontend JS errors, uncaught exceptions, and server-side errors per tenant. Tools like Sentry/Datadog can record stack traces with context (user/session IDs). Track the frequency of crashes and error types. High crash or exception rates indicate broken features.  
- **Rage Clicks (Frustration Signals):** Detect rapid repeated clicks on the same element (3+ clicks in seconds). A rage-click pattern (“frustration click”) usually means the UI is unresponsive or confusing. Logging these signals (via session replay or custom logic) helps pinpoint UX issues that ordinary analytics miss.  
- **Resource Usage:** Monitor infrastructure metrics by tenant (CPU, memory, DB queries). Identify “noisy neighbors” – tenants whose heavy usage affects others. Track database query duration and cache hit rates. Alerts on resource spikes can prevent degraded performance.  

## Data Minimization & Privacy Compliance  
- **Avoid Storing Raw PII:** By default, do **not** collect sensitive personal data (full names, emails, physical addresses, credit card numbers, social security numbers, health info, raw passwords, etc.) unless absolutely needed. Regulations (GDPR, CCPA, HIPAA, etc.) require *data minimization* – only collect data necessary for service. For any needed PII, apply strong protections (encrypt or tokenize it, restrict access).  
- **Mask Input Fields:** Any free-text fields or form inputs (especially passwords, financial data, emails) should be excluded or masked in analytics/session logs. For example, analytics SDKs (Amplitude, Pendo, Datadog) can auto-mask all form inputs or sensitive fields so that visitor session replays show asterisks instead of actual input. This ensures you don’t inadvertently store passwords or confidential text.  
- **Anonymization/Pseudonymization:** When possible, use anonymous or pseudonymous IDs instead of real user IDs. For example, store a hashed user ID or internal UUID rather than an email. Remove or hash IP addresses. Apply IP anonymization and truncate geolocation data if not needed.  
- **Consent & Rights:** If applicable, obtain user consent for tracking (per cookie laws). Provide mechanisms for users to view/export/delete their data (“right to be forgotten”). Ensure data retention policies are documented: **GDPR mandates** that personal data be kept “no longer than necessary” for its purpose, and any retention period must be justified and disclosed. Practically, many SaaS retention schedules delete raw logs after 1–3 years unless a longer justification exists.  
- **Regulatory Alignment:** Align with relevant frameworks: e.g. GDPR (EU users), CCPA/CPRA (California), HIPAA (if handling health data). Implement data-at-rest encryption, TLS in transit, and audit trails. List all data processing activities. Under GDPR, follow principles of lawfulness, purpose limitation, and security.  

## Multi-Tenant Security & Data Architecture  
- **Tenant Isolation:** Architect data stores so each tenant’s data is logically separated. Options include separate databases per tenant, separate schemas, or a shared schema with **strict row-level access controls**. In a shared-database model, every query must filter by `tenant_id` or use database row-security so no tenant can see another’s rows. Many scaled SaaS use a hybrid: e.g. free tiers share tables, mid tiers get separate schemas, and enterprise customers get dedicated databases. Always include `tenant_id` in your event logs and enforce it in every query or API call.  
- **Access Control & AuthN/Z:** Use strong authentication (e.g. SSO/MFA for admin users) and granular **role-based access control (RBAC)**. Grant the principle of least privilege: internal teams or apps see only the tenants’ data they need. For data access (e.g. embedded analytics), bind each user’s session to a JWT or token that includes their tenant (and role) to enforce row-level security. Audit all admin actions and data access by user/tenant.  
- **Encryption & Keys:** Encrypt data at rest and in transit. Use managed KMS solutions (AWS KMS, Azure Key Vault, HashiCorp Vault) to manage encryption keys. This protects against data breaches and satisfies PCI/GDPR/CCPA. For example, all database disks should be encrypted, and HTTPS/TLS should protect all API traffic. Use field-level encryption for ultra-sensitive PII if stored.  
- **Logging & Monitoring:** Log security events (failed logins, permission changes, data exports) with tenant and user context. Monitor logs centrally (e.g. Splunk, Datadog) to detect anomalies (e.g. a user from Tenant A trying to query Tenant B’s data). Set up alerts for unusual patterns (sudden spike in error rates or data access). Keep separate audit trails for each tenant to aid forensic analysis.  
- **Infrastructure & Networking:** Use network security (firewalls, VPCs) to segregate environments. Ensure your telemetry pipeline can tag events by tenant before storage. If using shared caching/message systems (Redis, Kafka), apply tenant-prefixed keys or topics so data never mixes. For example, use keys like `tenant:{tenantID}:session:{sessionID}` in Redis. In short, extend tenant isolation across all layers – databases, caches, queues and logs.  

## Event Schema & Retention Policies  
- **Tracking Schema:** Define a clear event schema. Every event should include a `tenant_id`, `user_id` (or anon user ID), `timestamp`, `event_type`, and any relevant properties (page, button_id, value). Design your schema so that joining across tables is easy (e.g. a Users table keyed by user_id within tenant). Avoid free-form or unbounded data fields.  
- **Data Retention:** Establish retention rules aligned with compliance. For example, anonymized aggregate metrics may be stored indefinitely, but raw event logs or PII-linked data might be purged after 1–2 years. Document retention in your data policies. Under GDPR/CCPA, once data is no longer needed for business/ legal purposes, it should be deleted. Implement automated data purge jobs. If a user or tenant requests deletion, ensure all their data (unless legally excepted) is removed or anonymized.  
- **Data Minimization & Aggregation:** Where possible, aggregate or roll up data. For example, store counts or metrics instead of raw events (e.g. daily active count per tenant instead of every login event long-term). This reduces sensitive data storage and speeds queries. For marketing analytics (conversion rates, funnel counts), storing anonymized identifiers or roll-ups often suffices.  
- **Compliance Checklist:** Maintain documentation of what data you collect and why. Ensure your privacy policy and terms reflect actual collection. Provide options for data subject requests (access, correction, deletion). Review that PII fields are hashed/omitted in analytics. Regularly audit code/SDKs to confirm no accidental logging of sensitive inputs. In short, enforce “privacy by design”: only track the metrics needed to run the business, and treat all user data (especially PII) as legally sensitive.
